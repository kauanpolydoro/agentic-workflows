# Ethical launch plan

## Pre-launch checklist

- Run the full acceptance suite from a clean clone.
- Review all compatibility and verification claims against committed evidence.
- Enable GitHub Pages, Issues, Discussions, private vulnerability reporting, and repository topics.
- Capture a short CLI demonstration using only included fixtures.
- Prepare the first tagged release without npm publication.

## Positioning

Agentic Workflows provides portable, inspectable procedures for coding agents, backed by strict metadata, safe local installation, and verification stages that do not confuse generated files with successful outcomes.

## Demonstration

Show catalog filtering, inspect `review-pull-request`, dry-run a generic installation, install it into a disposable repository, modify the generated file, and demonstrate that update preserves the modification.
Finish by showing the verification matrix and its Untested external-agent columns.

## Channel-specific publication

- Hacker News: “Show HN: Portable workflows for Claude Code, Codex, Cursor and other coding agents”.
- Reddit: choose one relevant developer-tool community, follow its self-promotion rules, and lead with implementation details and limitations.
- LinkedIn: explain the verification distinction and invite workflow-author feedback.
- X: use the short product statement and one link without engagement automation.
- Dev.to or Hashnode: publish a technical walkthrough of schema, adapters, and overwrite protection.

Do not post identical copy indiscriminately, buy or trade stars, reward stars, automate engagement, or conceal affiliation.

## Relevant communities

Consider Hacker News, language-specific TypeScript communities, open-source maintainer forums, and vendor communities only where project sharing is permitted.
Adapt every submission to local rules and audience needs.

## First four weeks

| Week | Focus | Release candidate |
| --- | --- | --- |
| 1 | Launch, onboarding feedback, CLI defects | Documentation and critical fixes |
| 2 | First fixture-backed external-agent evidence | Verification-only minor release |
| 3 | Community recipe authoring | Contributor-experience improvements |
| 4 | Review retention and support load | Prioritized catalog release |

Use weekly releases only when they contain reviewed value.

## Metrics and feedback

Track unique issue authors, successful clean-clone validations reported voluntarily, accepted recipes, time to first maintainer response, recurring failure categories, documentation corrections, and release downloads once releases exist.
Do not add product telemetry.

Convert feedback into issues by recording the user job, evidence, impact, acceptance criteria, and verification path.
Avoid promising priority or dates before maintainer review.
