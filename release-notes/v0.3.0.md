# Agentic Workflows v0.3.0

Agentic Workflows v0.3.0 adds an explicit autonomous execution-mode contract, the first unattended workflow design, and catalog discovery for execution mode.

## Highlights

### Autonomous workflow design

Recipe schema version 4 separates the engineering domain from the expected interaction model through `execution_mode: supervised | autonomous`.

Autonomous recipes must declare an upfront authorization model, no required mid-run human input, an owner-observable stop signal, a hard deadline, durable checkpoints, resumable execution, and an explicit failure policy.

The label describes the recipe design only.
It does not prove that an agent host can remain active, coordinate subagents, persist state, or complete the workflow.

### Autonomous GitHub issue campaigns

The new `resolve-github-issues` recipe defines a finite unattended campaign that prioritizes an authorized GitHub issue cohort and resolves the maximum safe number of eligible issues.

It distinguishes demonstrated non-defects and duplicates from functional, UX, accessibility, reliability, security, and compatibility problems.

Each code correction uses an isolated branch and worktree, commits, a pull request, a fresh adversarial review subagent, exact-head validation, direct squash merge, post-merge smoke verification, manual issue closure, and OID-bound branch cleanup.

The campaign uses bounded concurrency, a serialized integration lane, durable fenced checkpoints, explicit stop and deadline handling, and terminal states that do not overstate backlog exhaustion.

### Execution-mode discovery

The CLI accepts `awf list --execution-mode supervised|autonomous`, prints execution mode in human-readable list and detail output, and exposes the field in JSON catalog records.

Shell completion supports the new option and values in Bash, Zsh, Fish, and PowerShell.

The web catalog provides the same facet and labels autonomous designs independently from their verification status.

## Added

- The `resolve-github-issues` autonomous recipe and its complete seven-file source bundle.
- The strict autonomous execution contract in recipe schema version 4.
- Persistent-execution, subagent-orchestration, and distributed-coordination capability identifiers.
- CLI, shell-completion, JSON catalog, and web-catalog execution-mode discovery.
- Authoring guidance, autonomous workflow documentation, and ADR 0003.
- Unit, integration, acceptance, completion, generation, and package-smoke coverage for the new schema and discovery surface.

## Changed

- Migrated all 20 existing recipes from schema version 3 to schema version 4.
- Classified every existing recipe as `supervised` and incremented its recipe patch version to preserve bundle identity.
- Extended generated schemas, catalogs, manifests, compatibility documentation, and packaged offline documentation with execution-mode data.
- Kept installation, external-agent execution, and human-reviewed outcome evidence independent from design classification.

## Breaking changes

Recipe metadata using `schema_version: 3` is rejected by v0.3.0.

Every recipe must use `schema_version: 4` and declare `execution_mode` explicitly.
Supervised recipes omit `autonomy`, while autonomous recipes provide the complete strict contract and declare `persistent-execution`.

Consumers that parse recipe or catalog records must accept the new required `execution_mode` field and the optional autonomous contract.

## Upgrade

Update the globally installed CLI after publication:

```bash
npm install --global @kauanpolydoro/agentic-workflows@0.3.0
```

Confirm the selected version and inspect execution modes:

```bash
awf --version
awf list --execution-mode autonomous
awf show resolve-github-issues
```

For a project-local or CI installation, pin the exact version:

```bash
npm install --save-dev --save-exact @kauanpolydoro/agentic-workflows@0.3.0
npx awf list --execution-mode autonomous --json
```

Existing installed bundles are not silently rewritten.
Exact bundles installed by v0.2.2 remain available to `awf status`, `awf manifest`, `awf update`, and `awf remove` through retained fingerprints for all 20 historical recipes and all six adapters.
Use `awf status`, inspect an update with `awf update <workflow-id> --dry-run --show-content`, and apply it only after reviewing the schema version 4 bundle.
Unknown or tampered manifest identities remain fail-closed, while modified managed files are reported as drift and require explicit `--force` after review.

## Requirements and compatibility

- Node.js 22 or newer is required.
- Normal CLI operation remains offline and does not send telemetry.
- Recipes remain data and documentation, not executable plugins or a background agent runtime.
- The package exposes both `awf` and `agentic-workflows`, with `awf` used throughout the documentation.
- The unscoped `agentic-workflows` package on npm is a different project.

## Verification boundary

Repository validation covers strict schemas, deterministic generation, local and packed installation, exact v0.2.2 lifecycle upgrades, CLI automation, shell completion, terminal acceptance, cross-platform lifecycle behavior, documentation, and release contracts.

External-agent execution and human-reviewed outcomes remain untested unless a separate retained record identifies the agent version, environment, source revision, and review disposition.

The v0.3.0 tag and npm publication remain blocked until the repository's two recorded human editorial review gates are completed against the final candidate digest.
