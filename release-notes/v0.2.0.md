# Agentic Workflows v0.2.0

Agentic Workflows v0.2.0 turns the CLI into a complete, guided workflow for discovering, reviewing, installing, and maintaining agent workflows.

This release also establishes a reproducible publication path for the npm packages and GitHub release artifacts.

## Highlights

### Guided onboarding

Running `awf` without arguments now presents actionable first-run help.

Run `awf init` in an interactive terminal to choose the default agent and installation target through a guided wizard.

Use `awf init --wizard` when prompts are wanted with redirected input, or use `--no-interactive`, `--agent`, and `--target` for deterministic automation.

### Safer project and installation lifecycle

The new `awf context` command explains exactly which project root was selected and why.

The CLI now provides complete dry-run plans, managed-file drift reporting through `awf status`, manifest inspection through `awf manifest`, and recovery diagnostics through `awf doctor`.

Lifecycle locks and abandoned transaction directories remain fail-closed and require explicit human verification before removal.

### Better shell and automation support

`awf completion` generates command-specific completion for Bash, Zsh, Fish, and PowerShell.

Versioned JSON schemas now cover CLI results and errors, and the public `@kauanpolydoro/agentic-workflows/output-contract` export lets automation validate them directly.

The `normalizeProjectContext` helper provides one canonical project-root shape across `context`, `status`, `doctor`, and `init` results.

### Verified package consumption

The release packages were exercised as real npm tarballs through local installation, global installation, `npm exec`, `npx`, and Bun.

The CI matrix covers Ubuntu, macOS, Windows, and Node.js 22 compatibility, including guided initialization, interruption safety, lifecycle operations, and native shell completion.

## Added

- Guided `awf init` setup with explicit interactive and non-interactive modes.
- Project-root discovery and audit output through `awf context`.
- Installation health and drift reporting through `awf status`.
- Manifest inspection through `awf manifest`.
- Bash, Zsh, Fish, and PowerShell completion through `awf completion`.
- Public schemas for versioned CLI JSON output and structured errors.
- Actionable recovery metadata for lifecycle conflicts and interrupted transactions.

## Changed

- Empty states, invalid workflow IDs, unsupported configurations, and fallback project selection now include actionable next steps.
- Dry-run output now exposes complete file plans and can include generated content for review.
- Package smoke tests now validate both unpublished workspace packages from local tarballs before registry publication.
- The npm package README now uses the GitHub repository README as its canonical source.
- Release automation now verifies package integrity, source archives, checksums, SBOM data, provenance, and resumable GitHub release assets.

## Breaking changes

No intentional breaking changes are included in v0.2.0.

Existing installations remain managed through their hash-bearing manifests, and modified files continue to require explicit reconciliation.

## Upgrade

Update the globally installed CLI:

```bash
npm install --global @kauanpolydoro/agentic-workflows@0.2.0
```

Confirm the selected version and inspect the current project before changing files:

```bash
awf --version
awf context
awf doctor
awf status
```

Review `awf update <workflow-id> --dry-run --show-content` before applying an update to an installed workflow.

## Requirements and compatibility

- Node.js 22 or newer is required.
- Normal CLI operation remains offline and does not send telemetry.
- The package exposes both `awf` and `agentic-workflows`, with `awf` used throughout the documentation.
- The unscoped `agentic-workflows` package on npm is a different project.

## Security

This release does not claim a specific vulnerability fix.

The CLI continues to treat recipes as untrusted data, prevents silent overwrite of unmanaged or modified files, validates target containment and symlink parents, and never executes recipe content.
